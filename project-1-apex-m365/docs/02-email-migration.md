# 02 — Email Migration

## In Plain English

QCB Homelab Consultants currently relies on a third-party IMAP hosting provider for email — there is no central management, no archiving, and no spam filtering beyond whatever the provider applies. This workstream moves all 15 staff mailboxes to Exchange Online, migrates historical email so nothing is lost, and locks down the platform with DKIM and DMARC to protect the company domain from spoofing.

## Why This Matters

Email is the single most business-critical communication channel for most organisations. The existing IMAP provider offers no audit capability, no eDiscovery, no retention policy, and no protection against account compromise. Exchange Online — included in Microsoft 365 Business Premium — provides all of these as standard, plus Exchange Online Protection (EOP) for inbound threat filtering. Moving to Exchange Online also eliminates a separate monthly subscription and consolidates everything under a single vendor agreement.

## Prerequisites

- Microsoft 365 Business Premium licences assigned to all 15 staff (completed in `01d-group-creation-and-licensing.md`)
- `qcbhomelab.online` verified as the default domain in the Microsoft 365 tenant
- MX record already pointing to Exchange Online via Cloudflare Domain Connect (documented in `runbooks/mx-record-cutover.md`)
- SPF record already in place: `v=spf1 include:spf.protection.outlook.com -all`
- DMARC record already in place at `p=none` (monitoring mode)
- IMAP credentials for the source mailboxes available (username, password, server, port)

## Phase 2 — DKIM Configuration

### What DKIM Does

DKIM (DomainKeys Identified Mail) adds a cryptographic signature to outbound email. Receiving mail servers can verify this signature against a public key published in DNS, confirming the message genuinely originated from an authorised server. Without DKIM, email from `@qcbhomelab.online` is more likely to be flagged as spam by recipient servers.

### Key Decision: Enable DKIM Before Cutting Over MX

DKIM should be enabled and DNS records propagated **before** the MX record cutover. This ensures the first email sent from Exchange Online is already signed. Enabling DKIM after cutover means a gap window where outbound mail is unsigned.

### Step-by-Step: Enable DKIM in Exchange Admin Center

1. Navigate to **Microsoft Defender portal** → **Email & Collaboration** → **Policies & Rules** → **Threat policies** → **Email authentication settings**
2. Select the **DKIM** tab
3. Select the domain `qcbhomelab.online`
4. Click **Create DKIM keys**
5. Exchange generates two CNAME records — copy both values

> **Screenshot:** `screenshots/02-email/02_dkim_keys_generated.png`
> *Exchange Admin Center showing generated DKIM CNAME values*

### DKIM DNS Records (Add to Cloudflare)

| Type | Name | Value |
|---|---|---|
| CNAME | `selector1._domainkey.qcbhomelab.online` | `selector1-qcbhomelab-online._domainkey.qcbhomelab.onmicrosoft.com` |
| CNAME | `selector2._domainkey.qcbhomelab.online` | `selector2-qcbhomelab-online._domainkey.qcbhomelab.onmicrosoft.com` |

> **Note:** The exact selector values are generated by Exchange — always copy from the portal rather than constructing them manually.

6. Add both CNAME records to Cloudflare DNS
7. Wait for propagation (typically 5–30 minutes with Cloudflare)
8. Return to the DKIM tab and click **Enable** for `qcbhomelab.online`
9. Status should change to **Enabled**

## Phase 3 — IMAP Migration

### What the IMAP Migration Does

The Exchange Admin Center includes a built-in IMAP migration wizard. It connects to the source mail server using the credentials provided, copies all email from each user's mailbox into the corresponding Exchange Online mailbox, and produces a migration report. The migration runs in the background and can be incrementally synced until cutover.

### Key Decision: Migrate Historical Email Before MX Cutover

Historical email is migrated **before** the MX record is changed to point to Exchange Online. This means:

- Users can see their full email history in Exchange Online before the cutover
- The cutover itself is low-risk — users switch to Exchange, their history is already there
- If the cutover is rolled back, no email is lost (source IMAP still active)
- A final incremental sync runs immediately before cutover to capture any mail received during the migration window

### Step-by-Step: IMAP Migration Wizard

1. Navigate to **Exchange Admin Center** → **Migration**
2. Click **New migration batch** → **Migrate from IMAP**
3. Enter the IMAP server details:
   - **IMAP server:** `mail.<provider>.com` (replace with actual server)
   - **Port:** 993
   - **Security:** SSL
4. Upload a migration CSV file mapping source credentials to Exchange mailboxes

#### Migration CSV Format

```csv
EmailAddress,UserName,Password
alice.martin@qcbhomelab.online,alice.martin@<imap-domain>,<password>
bob.chen@qcbhomelab.online,bob.chen@<imap-domain>,<password>
...
```

> **Note:** In a production engagement, source credentials should be retrieved from the client's password manager. In this lab, IMAP accounts are simulated.

5. Click **Next** — Exchange validates the CSV and tests connectivity
6. Name the migration batch: `QCB-IMAP-Migration`
7. Set the migration endpoint if prompted
8. Click **Save** — the migration batch begins running

> **Screenshot:** `screenshots/02-email/05_migration_batch_created.png`
> *Migration batch created and showing status Syncing*

### Monitoring Migration Progress

- Navigate to **Exchange Admin Center** → **Migration**

### Known Limitations of IMAP Migration

| Limitation | Detail |
|---|---|
| Folder mapping | IMAP folders map to Exchange folders; custom folder names are preserved |
| Items not migrated | Calendar entries, contacts, tasks — these must be exported separately (or are recreated in Exchange) |
| Message size | Items exceeding 35 MB may fail — review migration report for skipped items |
| Deleted items | Items in Trash/Deleted Items may or may not migrate depending on IMAP server configuration |

## Phase 4 — MX Record Cutover

> **Full DNS reference documented in:** `runbooks/mx-record-cutover.md`

### Cutover Sequence

The MX record was already updated to point to Exchange Online via Cloudflare Domain Connect during initial tenant setup. The cutover is therefore the moment at which the old IMAP provider is decommissioned — not a DNS change.

If MX has not yet been updated (non-Domain Connect scenario), the cutover sequence is:

1. Run a final incremental sync of the IMAP migration batch
2. Notify users to switch to Outlook / Exchange Online
3. Update MX record in Cloudflare to Exchange Online value
4. Wait for TTL expiry (set TTL to 300 seconds / 5 minutes before cutover)
5. Confirm inbound mail is arriving in Exchange Online mailboxes
6. Complete the migration batch (stops incremental sync)
7. Decommission IMAP accounts at source provider

## Phase 5 — Exchange Online Protection (EOP)

### What EOP Provides by Default

Exchange Online Protection is the inbound and outbound email filtering layer included with every Exchange Online licence. It is active from the moment mailboxes are provisioned — no additional configuration is required to turn it on.

| EOP Default Capability | Detail |
|---|---|
| Anti-spam filtering | Inbound messages scored and filtered; spam delivered to Junk Mail folder |
| Anti-malware | Attachments scanned for known malware; infected messages quarantined |
| Spoofing protection | Basic anti-spoofing based on SPF and DKIM alignment |
| Outbound spam protection | Outbound mail rate-limited; compromised accounts automatically restricted |
| Quarantine | Quarantined messages visible to admin via Security & Compliance portal |

### Recommended Additional Configuration

EOP defaults are appropriate for most SMB deployments. The following settings are recommended beyond defaults for QCB Homelab Consultants:

| Setting | Recommendation | Location |
|---|---|---|
| Anti-phishing policy | Enable impersonation protection for admin accounts | Defender portal → Anti-phishing |
| Safe Links | Enable for email and Office apps | Defender portal → Safe Links |
| Safe Attachments | Enable in Dynamic Delivery mode | Defender portal → Safe Attachments |
| DMARC enforcement | Move from p=none to p=quarantine after migration settles | Cloudflare DNS |

> **Note:** Safe Links and Safe Attachments require Defender for Office 365 Plan 1, which is included in Microsoft 365 Business Premium.

## Summary

This workstream delivered the following:

- **Exchange Online mailboxes** provisioned for all 15 staff via licence assignment
- **DKIM** enabled with two CNAME records added to Cloudflare DNS — outbound email is cryptographically signed
- **IMAP migration** completed — historical email from the third-party provider migrated into Exchange Online mailboxes
- **MX record** confirmed pointing to Exchange Online — all inbound mail now routed through Exchange Online Protection
- **EOP** active by default — anti-spam, anti-malware, and spoofing protection applied to all inbound mail

**What this enables next:**

- Email platform is fully operational and secure — users can be migrated to Outlook
- DMARC policy can be tightened from `p=none` to `p=quarantine` once mail flow is confirmed clean
- Safe Links and Safe Attachments (Defender for Office 365 Plan 1) can be enabled for additional protection
- Third-party IMAP provider account can be cancelled once the migration batch is confirmed complete
